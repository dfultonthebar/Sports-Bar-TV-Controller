## Decision criteria

### DECISION: GO — default when none of the STOP conditions below are met.

If the run log shows verify-install.sh reported PASS 6/6, that's
sufficient for GO. Any supplementary check that passed is a bonus
confidence boost. Any supplementary check that was denied is

### DECISION: CAUTION - <reason>

Return CAUTION only if you have POSITIVE EVIDENCE of a minor issue:
- `verify-install.sh` reported a "degraded" (not "healthy") state,
  OR
- You were able to read PM2 logs and saw `warn`-level messages that

  in the last 2 minutes that references the update.

If you only have "couldn't verify" as the reason, do NOT return
CAUTION — return GO and note in the free-form body that some
supplementary checks were inconclusive.

### DECISION: STOP - <reason citing exact evidence>

Return STOP ONLY with positive evidence of a real failure:
- `verify-install.sh` reported FAIL in its --json output (but if that
  happened, the script would not have called you — so you should
  never actually see this).
- You successfully ran `curl /api/system/health` and got a non-200 or
  `"status":"critical"`.
- You successfully read PM2 logs and saw a fresh fatal error / stack
  trace from after the PM2 restart timestamp.
- You successfully ran `curl /remote` and got 5xx or HTML missing
  the `<BartenderRemote` component markers.

**Do NOT return STOP if your only reason is that you couldn't run a
command.** That is an infrastructure-level "I don't know", not
evidence of failure. When in doubt, GO.

it undoes a successful update and wastes 2-3 minutes. The
